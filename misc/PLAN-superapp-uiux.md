# AIOS Superapp — UI/UX Redesign, Polish & New-Features Plan

> scope: a full-app UI/UX pass over all ten surfaces, plus cross-cutting fluidity and the signature features that turn a working agent shell into a superapp — calm, consistent, and unmistakably yours.

## Executive summary

the app already does a great deal right under the hood, yet it reads as unfinished on first contact. seven themes explain why — and where the leverage is.

1. **the design system is documented but unadopted.** DESIGN.md sets real rules (accent discipline §6, the `.surface-pop` / `.pane-header` signatures, focus rings, token-driven color) and almost no surface honors them. composer surfaces hand-roll `.surface-pop`/`.focus-accent` with the wrong shadow (§4), the chrome strip and account badges use raw `text-white` instead of tokens (§7), browser/cast menus are ad-hoc `shadow-2xl` (§11), and xterm/Monaco themes are frozen hex that ignore accent and theme entirely (§12). the gap between the written system and the shipped pixels is the single biggest source of the "unpolished" feeling — and the cheapest to close.

2. **there is no front door, and the app greets you as a stranger.** no first-run onboarding exists at all — users cold-drop into the grid with no `onboardingComplete` flag and no veteran back-fill (§5). worse, the app ships someone else's identity: a `faeez` default, a hardcoded `firaz` greeting on the homescreen, a stranger's macOS home path seeding the money-agents fleet, and a dead-end account row (§5, §6, §13). identity is triple-sourced and Settings lies about wiring it (§13). the first impression is literally another person's account.

3. **the chat hero leaks telemetry before you've typed a word.** the empty new-chat state exposes the full telemetry stack, hard-cuts in with no entrance, says "starting claude" to a Codex user, dresses resting context chips in accent (failing the §6 litmus), and hides the real controls behind an unlabeled wrench — then hands you a blank box after asking "do anything" (§4). the screen where discovery matters most is the one doing the least to guide.

4. **accent is used as decoration, not meaning.** orange leaks everywhere it shouldn't: resting chips (§4), list bullets and armed drop-zones (§7, §9), generic border-hovers across all four superapp panes (§11). when everything is accented, nothing is — the eye loses the signal that accent is supposed to carry.

5. **the app is built for macOS and breaks on its actual platform.** Windows users are told to press ⌘K on the homescreen and throughout the palette; ~85 ⌘/⇧/⌥ literals across 15 files render wrong glyphs with no `platform.ts`; and Cast & Attach are macOS-only with no gating, so daily Windows users hit dead or confusing panes (§6, §8, §11, §13). the provider base compounds it: it's hardcoded to codex everywhere, and the migration actively downgrades claude users to codex on every load (§13).

6. **motion is missing, so the app feels static and abrupt.** heroes and transcript blocks pop in with zero entrance; pane mount/unmount and grid reflow have no motion; autoscroll snaps; the splash always shows and pops out with no exit; modals enter but don't leave; the clock snaps digits at 1Hz (§4, §6, §7, §9, §13). reduce-motion isn't even applied at boot, and the pet ignores it (§12, §13). the absence of choreography is felt as cheapness even when individual screens are fine.

7. **the superapp panes are powerful but untapped.** runEvents are already captured but never surfaced as a timeline; cast can mirror but never control; the palette writes a dead usage loop with no MRU; Mission Control silently no-ops at zero panes; panes can only reorder via two tiny arrows (§4, §7, §8, §9, §11). the boombastic features are mostly wiring up data the app already has.

> the good news: almost none of this is a rewrite — it's adoption, single-sourcing, and choreography over machinery that already works.

## How this document is organized

§1 reconciles this plan against the existing design system and audit findings. §2 defines the cross-cutting fluidity and motion baseline every surface inherits. §3 catalogs the new superapp features. §4–§13 are the per-surface plans (new-chat, onboarding, homescreen, shell, discoverability, transcript, secondary-pane convergence, superapp features, interactive surfaces, cross-cutting). the **roadmap** below sequences all of it into four tiers, and the appendix collects tokens, lint rules, and the platform map.

## Guardrails — the codebase author's locked decisions

> These come from the codebase author (firaz) via [POLISH-PLAN.md](POLISH-PLAN.md) and [HANDOFF-SESSION.md](HANDOFF-SESSION.md). They are load-bearing architectural decisions on the `windows-port` branch. This plan is designed to layer **on top** of them — every recommendation below is compatible with them. If you (driving this polish pass) want to override one, do it deliberately, not by accident.

1. **Four first-class panes only — chat · terminal · files · browser.** Everything else (notes, pet, money-agents, oracle, cast/mirror/attach) is optional and hidden-by-default behind ⌘K. Don't promote new panes to first-class. *(scope locked 2026-06-06.)*
2. **`TerminalComposer.tsx` is untouchable** — firaz: *"it's amazing already."* Port patterns **from** it; never edit it. Where this plan says "converge composer chips onto `.pill`," it means **ChatPane's** composer, not TerminalComposer.
3. **Keep the chat lean — no per-turn context bloat.** The per-turn preamble (`buildAiosShellContext`) was deliberately deleted: it re-injected lines every turn, induced hallucinated tool-talk, and dumbed the model. Do **not** reintroduce per-turn context injection. Session identity belongs in `CLAUDE.md`/`AGENTS.md`, read once via cwd. Any "smart" chat feature in §3 must route through the pane/command bus, **not** the model's context window.
4. **codex must feel identical to claude.** Engine parity is a hard requirement — every chat affordance works the same on claude, codex, and opencode. Codex/opencode deliver **whole replies** (no token streaming) by design; the "Working… m:ss" timer covers it — don't fake per-token streaming for them. (§2's streaming-caret motion is claude-only; the others get a graceful whole-message entrance instead.)
5. **ChatPane ↑-history recall stays as is** — firaz relies on it; don't redesign it.
6. **Build & verify from the terminal, never in-app** — rebuilding inside the app's own chat pane kills the live session. (Affects how you test, not the plan's content.)
7. **Provider decisions are locked:** ChatGPT via Codex native; opencode is the "everything-else / free fallback"; exactly one free model, no Chinese models. The §13 provider-base sweep makes the *base* follow the user's chosen CLI — it does **not** re-open these engine choices.

## §2. Fluidity & motion — making it feel sublime

### Motion principles (the preamble)

Five rules govern every animation in AIOS. They exist because the app is a calm, content-first workspace (DESIGN.md §1), and motion must reinforce that calm, never fight it.

1. **Calm, not busy.** Motion clarifies *where a thing came from and where it went*. It is spatial wayfinding, not entertainment. If an animation doesn't help you understand a state change, delete it.
2. **Fast.** Most transitions live in the **120–220 ms** band. Anything a user triggers and waits on (hover, press, menu open) must complete in ≤ 200 ms or it reads as lag. Reserve longer durations (300–600 ms) for ambient/decorative loops only.
3. **Transform & opacity only.** Animate `transform` and `opacity` for anything that moves or fades. Never animate `width`/`height`/`top`/`left`/`margin` on the hot path — they trigger layout and stutter. (Today the diff-expand at `ChatPane.tsx:4093` and tool-card reveals just hard-swap content with zero transition — see choreography below.)
4. **One ease language.** The app already converges on `cubic-bezier(0.16, 1, 0.3, 1)` (used by `.modal-in`, `.aios-fade-in`, `.aios-cell`, `.aios-ring`, `.aios-tile` in `App.css:298,316–318,342`). We promote that to a named token and standardize on it everywhere, so every surface decelerates with the same "settle."
5. **Reduce-motion is sacred.** Everything below collapses to instant under `[data-reduce-motion="true"]` (set on `:root` by `Settings.tsx:317`) **and** `@media (prefers-reduced-motion: reduce)`. The token layer is built so a single guard turns the whole system off.

### 2.1 The motion token layer — ADD to `App.css` `@theme`

These are purely additive (same convention as `--aios-radius-*`). Drop them inside the existing `@theme` block in `App.css` (after the `--aios-shadow-pop` line at `App.css:115`):

```css
/* ── motion: durations ───────────────────────────────────────
   The app's timing language. Most user-triggered motion is fast/base.
   slow/slower are for entrances + larger surface moves only. */
--aios-dur-instant: 80ms;   /* micro feedback: press, checkbox, dot */
--aios-dur-fast:    140ms;  /* hover, color/border, pill toggle, icon-btn */
--aios-dur-base:    200ms;  /* the default — menus, dropdowns, tab cross-fade */
--aios-dur-slow:    320ms;  /* pane mount, maximize/restore, modal in */
--aios-dur-slower:  480ms;  /* large entrances, route changes, list reveal */

/* ── motion: easings ─────────────────────────────────────────
   --aios-ease-out  : the house "settle" (already used app-wide). Default
                      for entrances + anything decelerating to rest.
   --aios-ease-in-out: symmetric, for things that move and return (drag snap).
   --aios-ease-spring: a gentle overshoot for playful affordances (send btn,
                      pane focus, tile lift). Use sparingly — it has personality.
   --aios-ease-in   : for EXITS only (accelerate away).  */
--aios-ease-out:    cubic-bezier(0.16, 1, 0.3, 1);
--aios-ease-in-out: cubic-bezier(0.65, 0, 0.35, 1);
--aios-ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1);
--aios-ease-in:     cubic-bezier(0.4, 0, 1, 1);

/* ── motion: composite shorthands (optional convenience) ────── */
--aios-trans-colors: color var(--aios-dur-fast) var(--aios-ease-out),
                     background-color var(--aios-dur-fast) var(--aios-ease-out),
                     border-color var(--aios-dur-fast) var(--aios-ease-out);
--aios-trans-pop:    transform var(--aios-dur-base) var(--aios-ease-spring),
                     box-shadow var(--aios-dur-base) var(--aios-ease-out);
```

**What each duration is for**

| Token | ms | Use |
|---|---|---|
| `--aios-dur-instant` | 80 | `.press` active state, checkbox/toggle flip, status-dot state change |
| `--aios-dur-fast` | 140 | hover (color/border/bg), `.pill` toggle, icon-button, `transition-colors` everywhere |
| `--aios-dur-base` | 200 | dropdown/menu open, command-palette panel, tab cross-fade, tool-card expand |
| `--aios-dur-slow` | 320 | pane mount, maximize⇄restore morph, modal/overlay in |
| `--aios-dur-slower` | 480 | list stagger total, route/section cross-fade, onboarding step |

> Migration note: existing hardcoded `0.22s` / `0.2s` / `120ms` durations in `App.css` (`.aios-tile` `App.css:342`, `.pill` `App.css:581`, `.aios-omni` `App.css:392`, resize handle `App.css:494`) should be retuned to reference these tokens so the whole app shares one clock. Tailwind `duration-*` utilities and `transition-colors` in `.tsx` files keep working but get a global baseline (§2.2) so even un-tuned ones ease consistently.

### 2.2 Global baseline + the single reduce-motion guard

Add this once to `App.css` (near the top, after the `html, body, #root` block at `App.css:208`). It gives **every** interactive element a default ease on the four "safe" properties, so nothing ever hard-snaps even if a component author forgot to add a transition — the #1 cause of "pop-in" today.

```css
/* Global motion baseline — every interactive surface eases color/border/shadow.
   Transform/opacity are NOT in the blanket rule (they'd animate things that
   shouldn't, e.g. scroll-driven transforms); opt into those via .lift/.press. */
a, button, [role="button"], [role="menuitem"], [role="tab"], summary,
input, textarea, select, .pill, .surface-card, .surface-pop,
[data-pane-key] {
  transition:
    color           var(--aios-dur-fast) var(--aios-ease-out),
    background-color var(--aios-dur-fast) var(--aios-ease-out),
    border-color    var(--aios-dur-fast) var(--aios-ease-out),
    box-shadow      var(--aios-dur-fast) var(--aios-ease-out),
    outline-color   var(--aios-dur-fast) var(--aios-ease-out);
}

/* THE master switch. One guard kills the whole motion system.
   Settings.tsx:317 already sets data-reduce-motion; we also honor the OS pref. */
[data-reduce-motion="true"] *,
[data-reduce-motion="true"] *::before,
[data-reduce-motion="true"] *::after {
  animation-duration: 0.001ms !important;
  animation-iteration-count: 1 !important;
  transition-duration: 0.001ms !important;
  scroll-behavior: auto !important;
}
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.001ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.001ms !important;
    scroll-behavior: auto !important;
  }
}
```

This **replaces** the long hand-maintained `[data-reduce-motion="true"] .brand-logo, .mascot-idle, …` allow-list at `App.css:459–473` (which only covers ~12 named classes and silently misses every new animation). The blanket rule is future-proof: any keyframe or transition you add is automatically governed. Keep the existing per-class `animation: none` blocks only where you want a class to *also* drop a `transform` (e.g. `.aios-tile--int:hover` at `App.css:367`), but the duration kill above is the safety net.

> Anti-stutter caveat: `0.001ms` rather than `0` so `transitionend`/`animationend` listeners still fire (some components rely on them to clean up).

### 2.3 Opt-in motion utilities — exact CSS

Add these as plain classes (same convention as `.pill`/`.surface-card`). They're the vocabulary every component reaches for instead of re-inventing motion.

```css
/* .lift — hover elevation for cards/tiles/rows. Transform+shadow only (GPU). */
.lift {
  transition: transform var(--aios-dur-base) var(--aios-ease-out),
              box-shadow var(--aios-dur-base) var(--aios-ease-out);
  will-change: transform;
}
.lift:hover { transform: translateY(-2px); box-shadow: var(--aios-shadow-pop); }
.lift:active { transform: translateY(0); transition-duration: var(--aios-dur-instant); }

/* .press — tactile push for buttons. Pairs with the global color baseline. */
.press { transition: transform var(--aios-dur-instant) var(--aios-ease-out); }
.press:active { transform: scale(0.96); }

/* .btn-glow — the ONE primary action (send btn). Accent ring breathes on hover.
   Accent-only per DESIGN.md §6; never put this on a neutral control. */
.btn-glow { position: relative; transition: var(--aios-trans-pop); }
.btn-glow:hover { transform: translateY(-1px);
  box-shadow: 0 0 0 4px color-mix(in srgb, var(--color-accent) 18%, transparent); }
.btn-glow:active { transform: translateY(0) scale(0.97); }

/* .fade-in-up — the canonical entrance. Replaces ad-hoc mounts.
   (.aios-fade-in at App.css:317 already does this 0.6s/8px — we keep that for the
   idle cockpit and add a faster general-purpose token-driven version.) */
@keyframes aios-fade-in-up { from { opacity: 0; transform: translateY(10px); }
                             to   { opacity: 1; transform: translateY(0); } }
.fade-in-up { animation: aios-fade-in-up var(--aios-dur-slow) var(--aios-ease-out) both; }

/* .scale-in — for popovers/menus growing from their trigger. */
@keyframes aios-scale-in { from { opacity: 0; transform: scale(0.96); }
                           to   { opacity: 1; transform: scale(1); } }
.scale-in { animation: aios-scale-in var(--aios-dur-base) var(--aios-ease-out) both;
            transform-origin: var(--aios-origin, top center); }

/* .stagger > * — list children cascade in. Set --n via inline style or :nth-child.
   contain:paint keeps each row's animation from invalidating siblings. */
.stagger > * { animation: aios-fade-in-up var(--aios-dur-base) var(--aios-ease-out) both; }
.stagger > *:nth-child(1) { animation-delay: 0ms; }

.stagger > *:nth-child(5) { animation-delay: 112ms; }
.stagger > *:nth-child(n+6) { animation-delay: 140ms; } /* cap so long lists don't crawl */
```

`will-change` discipline: `.lift` declares `will-change: transform` because hover is short-lived and predictable. **Do not** add `will-change` to perpetually-mounted surfaces (panes, the transcript) — a permanent `will-change` holds a GPU layer forever and *causes* jank. For `.fade-in-up`/`.scale-in`/`.stagger` we omit `will-change` and rely on the browser promoting during the animation only.

### 2.5 Specific anti-stutter guidance (the 60 fps budget)

- **Transform & opacity only on the hot path.** Every animation above moves things with `transform`/`opacity`. The two height animations (`.disclose`, sidebar width) use the `grid-template-rows: 0fr/1fr` trick or are one-shot/low-frequency — never animate `height`/`top`/`margin` on something that animates often. The current diff-expand (`ChatPane.tsx:4093`) and tool cards cause layout jumps precisely because they swap content with no transform-based reveal.
- **`will-change` discipline.** Add `will-change: transform` *only* to `.lift` (short hover) and the maximize-morph element *during* the morph (remove it `onAnimationEnd`). **Never** put `will-change` on `[data-pane-key]`, the transcript, or any always-mounted surface — a permanent compositor layer per pane will *cause* the jank you're trying to avoid, especially with embedded webviews already eating GPU.
- **Avoid layout thrash in JS.** The maximize FLIP and drag-ghost must read `getBoundingClientRect()` **once** before mutating, then only write `transform` (read-then-write, never interleaved). `PaneCard` already caches a rect via `wrapRef` (`App.tsx:3685`) — reuse it for the FLIP rather than re-measuring.
- **Virtualize / `contain` long lists.** The chat transcript, notifications, and file lists grow unbounded. Add `content-visibility: auto` + `contain-intrinsic-size` to off-screen turn rows and list items so streaming a long conversation doesn't repaint the whole DOM. Put `contain: paint` on `.stagger > *` and tool cards so one row animating can't invalidate its siblings.
- **Cap the stagger.** `.stagger`'s `:nth-child(n+6)` cap (above) ensures a 200-item list doesn't take 5 s to cascade — only the first ~5 visible rows animate; the rest are instant.
- **Composite-friendly properties only:** `transform`, `opacity`, `filter` (sparingly — `blur()` is expensive; the flash-aurora at `App.css:698` already uses `filter: blur(24px)` and is correctly gated to the `max` flash level + reduce-motion). Prefer `box-shadow` ring animations only on one-shot interactions, not infinite loops (an infinitely animating `box-shadow` repaints every frame — the existing `aios-dot-pulse`/`flash-breathe` loops are acceptable because they're tiny elements; don't scale that pattern to large surfaces).
- **One ease, one clock.** Because every duration/ease now flows from §2.1 tokens, retuning the whole app's "feel" is a five-line change, and nothing drifts out of sync. Audit-and-replace the hardcoded `0.2s`/`0.22s`/`120ms`/`160ms`/`300ms` literals in `App.css` (`.aios-tile` `:342`, `.pill` `:581`, `.aios-omni` `:392`, `.pet-starter-card` `:821`, `.pet-action-btn` `:790`, `.pet-meter-fill` `:1895`, resize handle `:494`) to the tokens.

### Repeat Last + Action Ring — instant re-fire of your recent moves

One-line hook: drag a pane and zones light up — drop it left/right/quadrant and the grid reflows to fit.

- **Why it's fun:** Today reordering is two tiny arrows. Letting you fling panes into snap zones is the single biggest "this feels like a pro OS" upgrade, and it exploits the grid you already have.

### Fluidity & motion

All entry/exit animations are gated under `[data-reduce-motion="true"]` via [`App.css:464`](src/App.css#L464) which already kills `.aios-fade-in` and `.aios-cell` — no new reduce-motion rules are needed as long as onboarding uses those existing classes.

**Card mount/dismiss.** Use the existing `.modal-in` class ([`App.css:298`](src/App.css#L298)): `aios-modal-in` runs `translateY(12px) scale(0.98) → identity` over `0.22s cubic-bezier(0.16, 1, 0.3, 1)` (the spring). Dismiss (skip or finish) plays the reverse inline: `animation: aios-modal-in 0.18s cubic-bezier(0.16, 1, 0.3, 1) reverse both`.

**Step transitions.** Do not unmount/remount the card between steps — that would re-trigger `.modal-in`. Instead, key only the step body on a wrapper: `<div key={step} className="aios-fade-in">…content…</div>`. `aios-fade-in` ([`App.css:304`](src/App.css#L304)) runs `translateY(8px) → identity` over `0.6s cubic-bezier(0.16, 1, 0.3, 1)` — with reduce-motion already killing it, no extra guard needed.

**Progress pips.** Each pip gets `transition: background-color 0.3s, width 0.3s ease` so the accent fill slides in as steps complete. The active pip can be 2px wider than inactive; animate width with the same `0.3s ease`.

**Provider status dots.** Render all rows immediately with a neutral `.status-dot` in `aios-dot-pulse` ([`App.css:285`](src/App.css#L285)) as the probing state. As each `detect_providers` result lands, crossfade the dot to `.status-dot--active` (green) or `.status-dot--cold` (muted) with a `0.3s color transition` on a small per-row stagger (40 ms × index). Do not hold rendering until all results arrive — the sequential reveal is the moment.

**MCP chip stagger.** When `<McpChips>` receives servers, apply `.aios-cell` ([`App.css:318`](src/App.css#L318)) with `animation-delay: calc(var(--i) * 40ms)` via a CSS custom property per chip. `aios-cell-in` runs `scale(0.4) → identity` over `0.4s cubic-bezier(0.16, 1, 0.3, 1)` — matches PluginsPane's existing visual language.

**Recheck spinner.** The 'recheck' button swaps its icon to `<RefreshCw className="animate-spin">` while `listPlugins` re-runs — mirrors the existing pattern at [`PluginsPane.tsx:61`](src/components/PluginsPane.tsx#L61).

**Greeting crossfade.** After onboarding writes `userName`, `IdleControlCenter`'s greeting wrapper already carries `.aios-fade-in` ([`IdleControlCenter.tsx:245`](src/components/IdleControlCenter.tsx#L245)). Force a remount by keying on `userName` so the real name fades in on first home render.

**AppearancePreview re-tint.** `setAccent`/`setTheme` already update CSS custom properties globally. Add `transition: background-color 0.2s, color 0.2s` to the swatch buttons inside `AppearancePreview` so the live re-tint glides rather than snaps.

### Fluidity & motion

The home already uses two motion primitives worth keeping and extending:

- **entrance** — `.aios-fade-in` ([App.css:317](src/App.css#L317)) uses `cubic-bezier(0.16, 1, 0.3, 1)` (the spring used throughout `.aios-ring`, `.aios-cell`, `.aios-drift-*`). - **focus rings** — the `focus-visible` ring on all launch-row controls should transition in via `transition-[box-shadow] duration-[120ms] ease-out` so tab-traversal glides rather than snapping; matches the `focus-within` shadow transition already on the `CommandLine` container ([IdleControlCenter.tsx:318](src/components/IdleControlCenter.tsx#L318)).
- **clock digits** — a 120–150 ms `opacity` ease (not a slide, which would feel busy) on the seconds glyph is enough to soften the 1 Hz tick without drawing attention to itself. Use a keyed `<span>` so React's reconciler knows to retrigger the transition.
- **ambient state** — the backdrop accent `color-mix` percentage should transition with `transition: --liveness 2s ease-in-out` on the outer container so agent-wake events breathe in slowly, never snap.
- **usage skeleton → content** — a 250 ms `opacity` crossfade from the shimmer placeholder to real `ProviderBlock` data. The skeleton itself should pulse at the same cadence as `aios-cadenced-shimmer` (if defined) or a simple `opacity: 0.4 → 0.7 → 0.4` at 1.8 s.
- **reduce-motion** — all of the above must be gated. `data-reduce-motion="true"` already kills `aios-drift-*`, `aios-fade-in`, and `aios-colon` ([App.css:464-472](src/App.css#L464)). Add the new `transition-[box-shadow]` and digit-opacity transitions to that block as `transition: none !important`.

### Fluidity & motion

all palette motion should use the existing `aios-modal-in` keyframe ([App.css:289-298](src/App.css#L289)) as the canonical enter template; exit is its mirror at `0.16s cubic-bezier(0.4,0,1,1)` (ease-in, intentionally faster than the `0.22s` open so the app snaps back to focus). new motion that does not already have a keyframe:

- **AI-intent row reveal (empty state):** `aios-fade-in` with an 8 px upward `translateY` over `0.6s cubic-bezier(0.16,1,0.3,1)` (the existing spring curve), staggered `40ms` per row. row heights should be pre-reserved so the mascot does not jump when the offer eases in.
- **MRU / recent group appear:** same `aios-fade-in` stagger when the `'recent'` group first renders; reserve the group-header height before the animation fires to prevent layout shift.
- **Selection accent bar:** a single `position:absolute` left-edge indicator that `translateY`s between rows using `0.12s cubic-bezier(0.16,1,0.3,1)`. this reads as the selection gliding — far more subliminal than each row independently flashing `bg-accent-soft`. `transition-colors` on the row background remains, but the bar is the primary affordance.
- **Cheat-sheet mount:** stagger the rows with `aios-fade-in` when the shortcuts section first mounts so the list cascades in rather than appearing as a static slab.
- **Smooth scroll for big jumps only:** `scrollIntoView({behavior:'smooth'})` exclusively for Page/Home/End; `'instant'` for single-step arrow to keep `↓`-held responsiveness crisp.
- **`[data-reduce-motion='true']` gate:** all of the above must be wrapped in the existing reduce-motion block ([App.css:460-473](src/App.css#L460)) — disable all transforms and durations, keep opacity transitions at `≤0.1s`.

### Fluidity & motion

the transcript is the highest-motion surface in the app, so every transition here must feel intentional rather than decorative.

**enter animation.** new blocks should enter with `aios-fade-in` (`App.css:317`): `translateY(8px) → translateY(0)` + `opacity 0→1` over 0.6 s with `cubic-bezier(0.16,1,0.3,1)` (the signature spring already in `App.css:304`). the class self-guards `@media (prefers-reduced-motion: reduce)` at `App.css:464` — nothing extra needed.

**expand/collapse.** all tool cards (`ActivityGroup`, `ActivityStep`, `ThinkingBlock`) currently do instant `{open && …}` mount/unmount. replace with `display:grid; grid-template-rows: 1fr` → `0fr` (inner div `overflow:hidden`), transition `200ms cubic-bezier(0.16,1,0.3,1)`. this also applies to the approval card height-collapse on resolution and the DiffBlock expand.

**error card.** the `AlertTriangle` should do a single 320 ms settle on enter (scale 0.85→1 + opacity, same spring easing) — error is a state, not a heartbeat, so no infinite pulse. retry button border lifts one step on hover over `120ms ease`.

**autoscroll.** user-initiated `jumpToLatest` uses native `behavior:'smooth'` (~300 ms). the jump pill itself ([`3654–3663`](src/components/ChatPane.tsx#L3654)) should enter with `aios-fade-in` rather than a hard appear. the streaming auto-pin stays instant on `rAF`-batched `scrollTop` to avoid lagging tokens; smooth only when `prefers-reduced-motion` is not set.

**find bar.** slides down from the top edge of the pane: `translateY(-8px) → 0` + opacity, `200ms cubic-bezier(0.16,1,0.3,1)`. match-highlight wash pulses once (`240ms`) on each next/prev navigation step.

**timestamps.** ride the existing `opacity-0 group-hover:opacity-100 transition-opacity` already present on the action row ([`4342`](src/components/ChatPane.tsx#L4342)) — no new animation budget needed.

all transitions above must be wrapped in or deferring to `@media (prefers-reduced-motion: reduce) { transition: none; animation: none; }` — the existing `App.css:464` block covers classes applied there; any inline style-based transitions need an explicit check against `window.matchMedia('(prefers-reduced-motion: reduce)')`.

### Fluidity & motion

all motion in this surface must be gated with `[data-reduce-motion='true'] { animation: none; transition: none; }` — match the guard already on `.status-dot--active` (`App.css:471`).

**cast/browser native-layer crossfade.** when `appcastStart` or `browser_show` resolves, animate the slot wrapper in with `transition-[opacity,transform] duration-300 cubic-bezier(0.16,1,0.3,1)` — the app's signature spring ease already used at `App.css:317`. start at `opacity-0 scale-[0.98]`, land at `opacity-100 scale-100`. the preceding placeholder should fade out simultaneously via `aios-fade-in` in reverse (`animation-direction: reverse`, 150 ms). this is the single highest-impact motion change in the section: a jarring snap becomes a quiet reveal.

**recording / mirroring pulse.** the danger dot at [AppCastPane.tsx:534](src/components/AppCastPane.tsx#L534) should use `animate-[aios-pulse_2s_ease-in-out_infinite]` (`App.css:283`). `aios-pulse` already scales 100 % → 130 % → 100 % with opacity; use it as-is but with `color-danger` fill. under `[data-reduce-motion='true']` the dot stays solid — no animation, but the `.pill` danger border still communicates live capture.

**surface-pop menus (picker + browser dropdowns).** enter: `scale-95 opacity-0 → scale-100 opacity-100`, 120 ms `ease-out`. exit: reverse, 80 ms. origin should be `transform-origin: top left` for the cast picker (which opens below-left of the trigger) so it grows out of the trigger rather than from center. use a CSS class pair (e.g. `menu-enter` / `menu-exit`) rather than inline style to keep the keyframe in one place.

**pane rows in MirrorViewer.** each `MirrorPaneRow` should enter with `aios-fade-in` (0.6 s, `App.css:317`) when it first appears in the snapshot. when the active pane changes, the active-pane card at [MirrorViewer.tsx:118](src/components/MirrorViewer.tsx#L118) should crossfade via a keyed `aios-fade-in` (use `key={active.key}`) so label changes don't hard-swap.

**toast slide-up.** both toasts ([AppCastPane.tsx:527](src/components/AppCastPane.tsx#L527), [BrowserPane.tsx:1493](src/components/BrowserPane.tsx#L1493)) should enter with `translate-y-1.5 opacity-0 → translate-y-0 opacity-100` (180 ms ease-out) and exit in reverse (120 ms). a single `toast-enter` keyframe in `App.css` shared by both is preferable to duplicating the inline style.

**new app cards (AttachAppsPane refresh).** newly-appeared rows after an interval refresh enter with `aios-fade-in`; existing rows must not reorder. stable sort by `bundle_id` before diffing ensures no positional jump.
